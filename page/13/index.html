第一次提交(21%, 39%)

思路：新建一个链表存储答案，遍历所有输入链表的头，选取最小值，复制到新链表中。

是纯暴力。

分析时间复杂度，选取kn个结点，每次用时k，为O(k2n)；空间复杂度为O(kn)

合并两个有序列表(LC题解)

简化问题，思考合并两个有序列表

为了达到空间代价是 O(1)，宗旨是「原地调整链表元素的 next 指针完成合并」

可以理解为用一根绳子把两行结点重新串起来，不用新建一行复制品

原地顺序合并（10%，90%）

思路：仍然需要遍历所有链表头找到最小值，但不新建链表复制最小值，而是原地将现有结点串起来。

空间复杂度为O(1)，时间复杂度仍然为O(k2n)

分治，原地合并（50%，80%）

思路：二分法合并，将问题分解到每次合并两条链表。

思考：用空间换时间，对比顺序合并，相当于将k个表头的对比（一次时间为k的循环）分解存储在树中。

k条链表，每条长度为n。

第一次合并后剩下 $ {k } $ 条，合并一条用时2n，故第一次合并一共用时 $ {k } * 2n$ = kn

第二次合并后剩下 $ {k } $ ，合并一条用时4n，故第二次合并一共用时 $ {k } * 4n$ = kn

···

二分合并一共需要合并log2k次，每次合并用时kn，故总时间复杂度为O(kn·log2k)

二分递归，树的高度为log2k，故空间复杂度为O(log2k)

吐槽：同样的代码用Java跑时间复杂度击败率可以到100%（

使用优先队列合并 （70%，30%）

思路：
仍然是原地合并
维护一个大小为k的小根堆，存放所有链表头
弹出堆顶连接，新表头进入堆

堆的的大小为k，故空间复杂度为O(k)

连接一个顶点的时间复杂度同等与维护一次堆logk，一共kn个结点。故时间复杂度为O(knlog2k)

平直着色与平滑着色

平直着色 将面根据面的法向进行着色，所以一个面上颜色不会有过渡，不同面因为法向不同颜色差别大。

平滑着色 一个面的每个地方着色时都会参考这个地方和面的所有点的法向，所以任意两个相近的地方的着色都会有圆滑的过渡。

平滑着色 会造成视觉上的棱角消失，但模型上的面没有增多，减少电脑负担。
有时候在同一件物体上需要保留其中某些棱角，抹平其它棱角。
方案一：打开自动光滑设置抹平棱角的范围，并打开法向加权修改
方案二：在编辑模式下部分着色

unorderer_map

简单来说，就是一个无序哈希表，时间复杂度O(1)

两个常用方法：

count()返回要查找的key在map的所有key中的出现次数。因为此容器不允许重复，故count()只可能返回 1 或 0，即可判断此key是否存在。有返回1，无返回0。
unordered_map也有find方法，得到的对象是一个iterator，在unordered_map中，如果find()没找到要找的key，就返回和end()一样的iterator值。

unorderer_set

unordered_set基于哈希表，也是无序的，但是，unordered_set是在哈希表中插入value，同时这个value也是它自己的key，和unordered_map不同，这里的key-value只是为了方便查阅，

两个常用方法：
同上，count（）和find（）

map和set区别

map：
map的值不作为键，键和值是分开的，所有元素都是键+值存在
不允许键重复
所有元素是通过键进行自动排序的
map的键是不能修改的，但是其键对应的值是可以修改的

set：
所得元素的只有key没有value，value就是key
不允许出现键值重复
所有的元素都会被自动排序
不能通过迭代器来改变set的值，因为set的值就是键

间接乘数：改变间接光的强度

阴影类型：硬阴影、软阴影（最耗费资源）、无阴影

绘制光晕

剔除遮罩：设定受灯光照射的图层

点光源

从一个点向球面发出光线，光强逐渐衰减至0

性能上，点光源比定向光源更耗费资源，尤其是在开启阴影的情况下

聚光源

锥形光，比点光源更耗费资源

区域光

从一个面发出的光，发出的光十分柔和，好的区域光可以很好的营造氛围

区域光不支持实时照明，只支持烘培照明

烘培

在窗口，渲染，照明中

对 拥有固定阴影的静态物体 绘制光照贴图，使其不参与阴影计算，节省大量性能

天空材质

太阳源：对象池中的某个光源
