Mirror基础总结（可跟随复习制作）

Posted on 2023-04-15 Edited on 2023-05-08

NetworkManager

是一个组件，多人联机的核心组件，创建一个空对象专门存储这个组件。

一个场景有且只能有一个Network Manager。Network Manager有DDOL选项，勾选以贯穿所有场景。同时Network Manager身上注册的所有生成预制体也会跟着到下一个场景生成。

设置离线和在线场景

NetworkManager自带的UI勾选上后可以直接使用，我写过自定义的连接UI
设置传输协议（Telepathy Transport）

Spawn
注册Player预制体
勾选自动创建Player
在连接上游戏后会自动创建联网的Player
该预制体需要有Network Identity。离线是将自动摧毁该对象
创建顺序和位置需要参考NetworkStartPosition结点（下面会讲）

在Registered Spawnable Prefabs注册后续可能在游戏中联网生成的其它对象（子弹、敌人），这些对象有netId。也可以写到这部分的时候回过头来注册，Spawn详细部分下面会讲。

在代码中ServerChangeScene方法可以改变场景，此时所有客户端都会跟着一起切换场景。ServerChangeScene的值会被更新。

常用API

属性名 说明
singleton The one and only NetworkManager（可以在别的脚本中通过NetworkManager.singleton调用）
numPlayers Number of active player objects across all connections on the server.
networkSceneName The name of the current network scene.

NetworkRoomManager

NetworkRoomManager是Network Manager的派生类，包括拥有后者的所有功能，因此不能共存。

请勾选上Show Room GUI，我还没写过自定义游戏大厅的UI和连接方式。（但是在NetworkRoomManager和Network Room Player的专门的博客里我有把相关方法和属性列进去）

将在游戏大厅中生成的玩家所需要用到的预制体称为大厅玩家预制体，相应的我将在真正游戏中生成玩家所需要用到的预制体称为游戏玩家预制体，两者独立。请专门另外做一种大厅玩家预制体，注册上。

Network Start Position

一个组件，创建一个结点专门放置这个组件最为一个出生点。
可以创建多个出生点，分布在场景各个地方。
场景加载完后，NetworkManager会在这些地方有根据的生成玩家。

Network Identity

组件
在联网游戏进行时所用到的预制体必须要有NetworkIdentity组件
加载场景顺序：
加载没有NetworkIdentity组件的结点
等待联网
当一个玩家联网时，加载（更新）所有最新状态的拥有NetworkIdentity组件的结点

运行逻辑与顺序

在本地客户端运行游戏，场景内会有多个人物，其中有一个人物由本地控制——称为玩家
另外几个人物也是本地生成，组件、材质、脚本都于本地运行，但它们的控制命令来自于服务器——称为克隆玩家
控制玩家实际上是向服务器发送行动请求，服务器许可后玩家及其对应的所有克隆玩家行动

Network Attributes

同步变量标记，被标记的变量会同步给其他玩家
那么一旦此变量(在服务器！！！！)发生变化，服务器就会将此变量发送给所有客户端
后面是同步变量被修改时的监听方法

Command标记是针对方法的标记，被标记的方法必须以Cmd开头
CmdSetupPlayer这个方法是被 [Command]标记过的，所以是会在服务器端执行的，不会再本地执行
如果取消了[Command] 的标志那就肯定只在本地执行了也就是完全的本地的方法啊和服务端没关系
在服务器中改变同步变量才会被分发给客户端！！！！！！！！！！！！！
Command方法描述为 会被客户端调用的服务端的方法

Bypassing Authority
为了安全起见, [Command] 只能在player object上使用
non-player objects想要使用[Command]需要拥有authority，或者使用[Command(requiresAuthority = false)]可以绕过权限检查（但无法传递参数）

关于标记更多信息参考以前写的博客Mirror_Network Attributes
